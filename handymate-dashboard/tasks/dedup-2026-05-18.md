## Architecture-beslut

### v1: applikations-härledd dedup_key
Strategy: `dedup_key = ${agent_id}:${knowledge_type}:${normalized_title}`
där `normalized_title` = lowercase + strip siffror/datum/procent.

**Varför härledd istället för agent-satt (v1)**: Vi vet inte hur observationerna
faktiskt ser ut än — vi har ett riktigt pilot-företag (biz_al7pjuu5smi) och tre
test-businesses utan data. Att prompt-instruera agenter att sätta semantisk
dedup_key (t.ex. `"stale_quote:${quote_id}"`) utan att veta hur de duplicerar
i praktiken = gissning. Härledd nyckel täcker majoriteten av fallen och vi
ser i pilot-data om den är otillräcklig.

**Varför applikationskod istället för PostgreSQL GENERATED STORED**: Stelhet.
Dedup-strategin kommer behöva tune:as efter pilot-data (per-typ-fönster, ev.
semantiska nycklar). Applikationskod kan ändras utan SQL-migration.

### v2 (post-pilot, om data motiverar): agent-satt dedup_key
`AgentObservation`-interface har redan `dedup_key?: string`-fält i v1. När
pilot visar att vissa observation-typer dupliceras dåligt (t.ex. Daniel skriver
"BRF Lindgården öppnat 3 ggr" söndag och "BRF Lindgården öppnat 4 ggr" onsdag
— olika normalized_title) kan vi prompt-instruera agenten:

> "För observations som handlar om specifika offerter/projekt, sätt
> `dedup_key: \"stale_quote:${quote_id}\"` så vi inte upprepar samma fenomen."

## Dedup-fönster

```ts
const DEDUP_WINDOWS_HOURS = {
  anomaly: 48,        // akut — OK med påminnelse efter 2 dagar
}
const DEFAULT_DEDUP_WINDOW_HOURS = 168  // 7 dagar
```

- `anomaly` (akuta, t.ex. projekt över budget) får 48h → om problemet kvarstår
  ny vecka kan agenten flagga igen.
- Allt annat (`pattern`, `insight`, `recommendation`) får 7 dagar default.
  Pattern-observationer (BRF accepterar 92%) är stabila över veckor.

Per-typ-fönster är **inte tune:ade på pilot-data** — det är TD-46.

## Flow per observation

```
1. agent kör (söndag eller onsdag cron)
2. Claude returnerar 1-3 observationer
3. för varje observation:
   a. computeDedupKey(agentId, observation)
      → om obs.dedup_key satt: använd den
      → annars: ${agentId}:${knowledge_type}:${normalizeTitle(title)}
   b. windowHours = getDedupWindowHours(knowledge_type)
   c. findRecentDuplicate(supabase, businessId, agentId, dedupKey, windowHours)
   d. om duplicate hittas:
      → skip INSERT business_knowledge
      → skip push
      → logga [dedup] skipped agent=X key=Y existing_id=Z
      → skippedDuplicates++
   e. om ingen duplicate:
      → INSERT business_knowledge (med dedup_key-kolumn)
      → om suggestion: INSERT pending_approvals + push 'agent_observation'
      → om ingen suggestion: bara push 'agent_insight'
4. saveAndPush returnerar:
   { saved, approvals_created, insights_pushed,
     skipped_duplicates, skipped_details: [...] }
```

## Manuell SQL — KÖR INNAN SÖNDAG 2026-05-22 06:00 UTC

Filen `sql/v2_business_knowledge_dedup.sql` måste köras manuellt i
Supabase SQL Editor (per CLAUDE.md-konvention).

```sql
ALTER TABLE business_knowledge
  ADD COLUMN IF NOT EXISTS dedup_key TEXT;

CREATE INDEX IF NOT EXISTS idx_bk_dedup
  ON business_knowledge(business_id, agent_id, dedup_key)
  WHERE status = 'active' AND dedup_key IS NOT NULL;
```

**Verifiering efter SQL-körning:**

```sql
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'business_knowledge' AND column_name = 'dedup_key';
-- → 1 rad: dedup_key | text | YES
```

**Konsekvens om SQL inte är körd**: saveAndPush kommer crasha på INSERT pga
saknad `dedup_key`-kolumn. Hela observation-pipelinen för agenten faller.

## Identifierade TDs

### TD-45: Trigger-driven observations för intra-vecka-events

**Vad**: Idag triggas observations endast via cron (söndag + onsdag 06:00-06:15).
Vissa observation-typer skulle ge mer värde om de genererades reaktivt:

- Karin checkar fakturera-möjlighet inom 1h efter Lisa tar samtal
- Lars flaggar samma dag att projekt går över budget
- Daniel pingar samma timme att offert öppnats 3:e gången

**Varför inte v1**: Behöver pilot-data först för att veta vilka events som
faktiskt är värda push-notiser. Risk för spam annars.

**När**: Post-pilot, baserat på engagement-data (klick-rate, dismiss-rate).

### TD-46: Per-knowledge_type-fönster baserat på pilot-data

**Vad**: Nuvarande fönster (`anomaly: 48h`, default: `168h`) är gissning utan
data. Efter 2-3 veckors pilot bör vi mäta:

- Hur ofta dedup utlöses per knowledge_type
- Hur ofta hantverkare engagerar med "samma" observation i två veckor i rad
- Om `pattern`-observationer bör ha 14d-fönster istället för 7d

**Datakälla**: `skipped_details`-loggar från saveAndPush + dashboard-engagement.

**När**: Efter 2-3 söndag-cron-runorna 2026-05-22, 2026-05-25 (onsdag),
2026-05-29 (söndag). Då har vi 4 körningar att jämföra.

### TD-47: dedup_key inkluderar inte data_basis-signal

**Vad**: v1 dedup_key härleds från `(agent_id, knowledge_type, normalized_title)`.
Om Daniel skriver "Stale offert hett intresse" söndag (avser BRF Lindgården) och
onsdag (avser Anders Eriksson) → samma dedup_key, andra körningen skippad
felaktigt.

**Mitigation v1**: Pilot-data visar om detta är vanligt. Om ja → v2 prompt-
instruktion att inkludera kund-namn/quote-id i titel ELLER agent-satt
dedup_key med entity-id.

**När**: Tune efter pilot-vecka 1-2.

### TD-48: Soft-dedup för observability under pilot

**Vad**: Idag hard-skippas duplicater (skip INSERT). Alternative: INSERT
med status='duplicate' så vi kan se duplicering-mönster i `business_knowledge`-
tabellen retrospektivt.

**Varför inte v1**: Skulle kräva uppdatering av frontend-queries att filtrera
bort `status='duplicate'`. Hard-skip + `skipped_details`-loggning räcker för
pilot-observability.

**När**: Om vi behöver mer detalj än cron-loggar visar.

## Begränsningar i v1

1. **Inga agenter sätter dedup_key själva** — alla går via härledd nyckel.
   Reserverat för v2 (TD-47).
2. **Per-typ-fönster är gissade** — TD-46 tune:as på pilot-data.
3. **Cron-routes har inte uppdaterats att inkludera `skipped_duplicates` i
   payload** — test-endpoint returnerar dem via direkt JSON-spread från
   `runXObservation`-funktionen. Cron-route loggar dem i Vercel-logg.
4. **Frontend visar inte dedup-stats** — `business_knowledge`-tabellen visas
   som vanligt utan hint om hur många duplicater som skippats. Inte blocker
   för pilot.

### BUG-001: Saknad error handling på .single() i invoices/new
**Fil:** `app/dashboard/invoices/new/page.tsx` rad 94
**Problem:** `.single()` på business_config-query kastas utan null-check. Om ingen konfiguration finns returneras error men felhantering saknas.
**Symptom:** Applikationen kraschar eller visar blank sida när användare saknar business_config
**Rotorsak:** Supabase `.single()` kastar exception om noll eller flera rader hittas, men exception fångas inte
**Fix:**
```typescript
// FEL:
const configRes = supabase
  .from('business_config')
  .select('default_payment_days, invoice_prefix, next_invoice_number, default_hourly_rate')
  .eq('business_id', business.business_id)
  .single()

// RÄTT:
const { data: configData, error: configError } = await supabase
  .from('business_config')
  .select('default_payment_days, invoice_prefix, next_invoice_number, default_hourly_rate')
  .eq('business_id', business.business_id)
  .single()

if (!configError && configData) {
  setDueDays(configData.default_payment_days || 30)
  setInvoicePrefix(configData.invoice_prefix || 'FV')
  setNextNumber(configData.next_invoice_number || 1)
} else {
  setDueDays(30)
  setInvoicePrefix('FV')
  setNextNumber(1)
}
```

### BUG-013: DELETE projekt raderar barnrader utan business_id-verifiering (cross-tenant risk)
**Fil:** `app/api/projects/route.ts` rad 454–480  
**Problem:** Barn-tabeller raderas med enbart `project_id`-filter INNAN business_id-verifieringen på `project`-tabellen körs. En business som känner till ett `project_id` från annan business kan radera deras dokument, loggar och checklistor.  
**Symptom:** Cross-tenant dataskada vid kännedom om foreign project_id.  
**Rotorsak:** Auth-verifiering sker sist i DELETE — barn-queries körs dessförinnan utan tenant-isolering.

```typescript
// FEL — verifiera ägarskap EFTER barn-deletes:
await supabase.from('project_document').delete().eq('project_id', projectId)
await supabase.from('project_log').delete().eq('order_id', projectId)
// ...
await supabase.from('project').delete().eq('project_id', projectId).eq('business_id', business.business_id)

// RÄTT — verifiera ägarskap FÖRST:
const { data: ownedProject } = await supabase
  .from('project')
  .select('project_id')
  .eq('project_id', projectId)
  .eq('business_id', business.business_id)
  .single()

if (!ownedProject) {
  return NextResponse.json({ error: 'Not found' }, { status: 404 })
}
// Nu är det säkert att radera barn
await supabase.from('project_document').delete().eq('project_id', projectId)
// ...
await supabase.from('project').delete().eq('project_id', projectId)
```

### BUG-007: Auth token parsing swallowear fel tyst
**Fil:** `lib/auth.ts` rad 59–68

**Symptom:** Användare får plötsligt 401 utan förklaring när auth-token är korrupt
**Rotorsak:** Error-handling döljer teknisk information
**Fix:**

```typescript
// FEL:
try {
  const decoded = decodeURIComponent(sbCookie[1])
  const parsed = JSON.parse(decoded)
  accessToken = parsed[0]

  // Ignorera parse-fel
}

// RÄTT:
try {
  const decoded = decodeURIComponent(sbCookie[1])
  const parsed = JSON.parse(decoded)

} catch (parseErr) {
  console.warn('[Auth] Failed to parse auth cookie:', parseErr instanceof Error ? parseErr.message : 'Unknown')
}
```

### BUG-016: /api/leads saknar try/catch i alla tre handlers
**Fil:** `app/api/leads/route.ts`
**Problem:** GET, PATCH och POST har inga try/catch-block. Om Supabase kastar ett undantag (nätverksavbrott, schema-fel, RPC-fel) avbryts funktionen med en unhandled promise rejection — Next.js returnerar en generisk 500-sida utan JSON-body.
**Symptom:** Pipeline-vyn laddas inte och visar nätverksfel utan förklaring. PATCH (statusändring) och POST (seed scoring rules) misslyckas tyst.
**Rotorsak:** Route-filen är troligen skriven utan det try/catch-mönster som alla övriga routes använder.

```typescript
// FEL — GET utan try/catch:
export async function GET(request: NextRequest) {
  const business = await getAuthenticatedBusiness(request)
  if (!business) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  // direkt Supabase-anrop utan skydd...
}

// RÄTT:
export async function GET(request: NextRequest) {
  try {
    const business = await getAuthenticatedBusiness(request)
    if (!business) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    // ...
  } catch (error: any) {
    console.error('Get leads error:', error)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
}
// Samma mönster för PATCH och POST
```

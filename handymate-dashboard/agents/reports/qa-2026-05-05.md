## Kritiska buggar

### BUG-001: Direkt Supabase-insert av `time_entry` från klient — kringgår all auktorisering
**Fil:** `app/dashboard/projects/[id]/page.tsx` rad 687
**Problem:** `handleTimeSave()` anropar `supabase.from('time_entry').insert(entryData)` direkt från klienten utan server-side business_id-kontroll. RLS-policy är enda skyddet.
**Symptom:** En inloggad användare kan potentiellt lägga tidrapporter mot vilket projekt som helst om RLS är felaktigt konfigurerad eller saknas.
**Rotorsak:** Time-entry-formuläret i projektsidan POSTar direkt via Supabase-klienten istället för via `/api/time` (som har `getAuthenticatedBusiness`-kontroll).
**Fix:**
```typescript
// FEL (rad 687):
const { error } = await supabase.from('time_entry').insert(entryData)

// RÄTT — använd API-route med autentisering:
const res = await fetch('/api/time', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(entryData),
})
if (!res.ok) throw new Error((await res.json()).error || 'Fel')
```

### BUG-002: `/api/leads` GET, PATCH och POST saknar try/catch — okontrollerade undantag kraschar routen
**Fil:** `app/api/leads/route.ts` rad 6 (GET), 88 (PATCH), 144 (POST)
**Problem:** Ingen av de tre handlers är insvept i try/catch. Om Supabase kastar ett oväntat undantag (nätverksfel, timeout) kraschar Next.js-routen utan ett strukturerat JSON-svar.
**Symptom:** Pipeline-sidan visar tom skärm eller evig spinner. Vercel-loggar visar unhandled promise rejection.
**Rotorsak:** Saknar `try { ... } catch (error: any) { return NextResponse.json({ error: error.message }, { status: 500 }) }`.
**Fix:**
```typescript
// RÄTT — omslut varje handler:
export async function GET(request: NextRequest) {
  try {
    const business = await getAuthenticatedBusiness(request)
    if (!business) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    // ... befintlig logik
    return NextResponse.json({ leads: ..., stats: ... })
  } catch (error: any) {
    console.error('Get leads error:', error)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
}
```

### BUG-003: `setInterval` i module-scope i `lib/auth.ts` — minnesläcka i serverless-miljö
**Fil:** `lib/auth.ts` rad 149–156
**Problem:** `setInterval` körs på module-level utanför en funktion. I Vercel serverless lever varje lambda-instans kortare tid än 5 minuter, vilket innebär att intervallet aldrig triggar och `rateLimitStore` aldrig rensas. I edge-fall kan det orsaka minnesläcka.
**Symptom:** Rate-limiting fungerar inte korrekt i produktion — counters ackumuleras utan att nollställas.
**Rotorsak:** `setInterval` är ett Node-konstrukt som inte passar serverless. In-memory rate limiting behöver lazy cleanup eller ersättas med Redis/KV.
**Fix:**
```typescript
// FEL (rad 149): setInterval på module-level — ta bort detta block.

// RÄTT — lazy cleanup inne i checkRateLimit:
export function checkRateLimit(key: string, config: RateLimitConfig) {
  const now = Date.now()
  // Rensa expired entries vid varje access istället för med interval
  if (rateLimitStore.size > 1000) {
    rateLimitStore.forEach((entry, k) => {
      if (entry.resetAt < now) rateLimitStore.delete(k)
    })
  }
  // ... resten av logiken oförändrad
}
```

### BUG-004: `fetchReports` i `field_reports`-subkomponent — `setLoading(false)` utanför finally
**Fil:** `app/dashboard/projects/[id]/page.tsx` rad 6447–6453
**Problem:** `setLoading(false)` anropas efter try/catch-blocket men inte i ett `finally`-block. Om `fetch` kastar vid nätverksfel körs catch med `/* silent */` och sedan `setLoading(false)` — detta fungerar idag men är ett skört mönster; om catch-blocket i framtiden utökas med `throw` fastnar spinnern.
**Symptom:** Potentiell evig spinner om catch-blocket ändras eller om ett synkront undantag kastas.
**Fix:**
```typescript
// FEL:
async function fetchReports() {
  try {
    const res = await fetch(`/api/field-reports?project_id=${projectId}`)
    const data = await res.json()
    setReports(data.reports || [])
  } catch { /* silent */ }
  setLoading(false)
}

// RÄTT:
async function fetchReports() {
  try {
    const res = await fetch(`/api/field-reports?project_id=${projectId}`)
    const data = await res.json()
    setReports(data.reports || [])
  } catch { /* silent */ }
  finally {
    setLoading(false)
  }
}
```

### BUG-005: Four-eyes-kontroll kringgås när `budget_amount` skickas i body
**Fil:** `app/api/projects/route.ts` rad 357–399
**Problem:** `projectValue = body.budget_amount || 0` — om anroparen skickar `budget_amount > 0` i body hoppas hela four-eyes-blocket över (`if (!projectValue)` är false). Projektet kan stängas utan godkännandeprocess oavsett värde.
**Symptom:** Projektstängning på t.ex. 200 000 kr kräver ingen godkännandeprocess om body innehåller `budget_amount`.
**Fix:**
```typescript
// FEL:
const projectValue = body.budget_amount || 0
if (!projectValue) {
  // hämta från DB
}

// RÄTT — hämta alltid från DB för att avgöra four-eyes:
const { data: existingProject } = await supabase
  .from('project')
  .select('budget_amount')
  .eq('project_id', project_id)
  .single()
const projectValue = existingProject?.budget_amount || body.budget_amount || 0
```

### BUG-006: `saveAsTemplate()` visar success-toast vid API-fel
**Fil:** `app/dashboard/quotes/new/page.tsx` rad 1185
**Problem:** `fetch('/api/quote-templates', {...})` — svarets `res.ok` kontrolleras aldrig. Om API:t returnerar 4xx/5xx visas ändå "Mall sparad!" och modalen stängs.
**Symptom:** Hantverkaren tror att mallen sparades men ingenting lagrades i databasen.
**Fix:**
```typescript
// FEL:
await fetch('/api/quote-templates', { method: 'POST', ... })
toast.success('Mall sparad!')

// RÄTT:
const res = await fetch('/api/quote-templates', { method: 'POST', ... })
const data = await res.json()
if (!res.ok) {
  toast.error(data.error || 'Kunde inte spara mallen')
  return
}
toast.success('Mall sparad!')
```

### BUG-007: `NewQuotePage.fetchData()` saknar try/catch — evig spinner vid API-fel
**Fil:** `app/dashboard/quotes/new/page.tsx` rad 563–596
**Problem:** `fetchData()` kör `Promise.all` utan try/catch. Om `/api/customers` returnerar fel eller Supabase kastar anropas aldrig `setLoading(false)` (rad 595).
**Symptom:** Ny-offert-sidan visar spinner för alltid om ett API-anrop misslyckas.
**Fix:**
```typescript
// FEL:
async function fetchData() {
  const [customersApiRes, settingsRes] = await Promise.all([...])
  // ...
  setLoading(false)  // körs bara vid lyckat genomlopp
}

// RÄTT:
async function fetchData() {
  try {
    const [customersApiRes, settingsRes] = await Promise.all([...])
    // ...
  } catch (err) {
    console.error('[NewQuote] fetchData misslyckades:', err)
  } finally {
    setLoading(false)
  }
}
```

### BUG-009: `auth.ts` — `getAuthenticatedBusiness` business_users-fallback använder `.single()` istället för `.maybeSingle()`
**Fil:** `lib/auth.ts` rad 93–110
**Problem:** `business_users`-fallback-flödet anropar `.single()` (rad 99) för att hämta business_config. `.single()` kastar `PGRST116` om ingen rad hittas, vilket swallas av den yttre catch och returnerar null utan logg — ger silent auth-failure.
**Symptom:** Anställda vars business inte hittas i business_config får ingen autentisering och inga loggar — svårt att debugga.
**Fix:**
```typescript
// FEL (rad 104):
const { data: employeeBusiness } = await supabase
  .from('business_config')
  .select('*')
  .eq('business_id', businessUser.business_id)
  .single()

// RÄTT:
const { data: employeeBusiness, error: empErr } = await supabase
  .from('business_config')
  .select('*')
  .eq('business_id', businessUser.business_id)
  .maybeSingle()
```

**Fil:** `app/dashboard/vehicles/page.tsx` rad 156–158

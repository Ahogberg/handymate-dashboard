## Kritiska buggar

### BUG-001: Infinite spinner om fetchData kastar i invoices/new
**Fil:** `app/dashboard/invoices/new/page.tsx` rad 82–115
**Problem:** `fetchData()` har ingen try/catch/finally. Om någon av de tre `Promise.all`-anropen kastar (Supabase-timeout, nätverksfel) anropas aldrig `setLoading(false)`.
**Symptom:** Användaren ser en oändlig spinner och kan inte skapa faktura.
**Rotorsak:** Asynkron funktion utan felhantering — undantaget swallowas tyst av React, `loading` fastnar på `true`.
**Fix:**
```typescript
// FEL:
async function fetchData() {
  const [customersApiRes, timeRes, configRes] = await Promise.all([...])
  // ...
  setLoading(false)
}

// RÄTT:
async function fetchData() {
  try {
    const [customersApiRes, timeRes, configRes] = await Promise.all([...])
    // ...
  } catch (err) {
    console.error('fetchData error:', err)
  } finally {
    setLoading(false)
  }
}
```

### BUG-002: Infinite spinner om fetchAll kastar i time/page
**Fil:** `app/dashboard/time/page.tsx` rad 186–188
**Problem:** `fetchAll()` anropar `Promise.all` med fem sub-funktioner utan try/catch. `setLoading(false)` anropas aldrig om något kastar.
**Symptom:** Tidrapporterings-sidan fastnar i laddningsskärmen.
**Rotorsak:** Samma mönster som BUG-001 — asynkron funktion utan finally.
**Fix:**
```typescript
// FEL:
async function fetchAll() {
  await Promise.all([fetchEntries(), fetchWorkTypes(), fetchCustomersAndBookings(), fetchStats(), fetchTeamMembers()])
  setLoading(false)
}

// RÄTT:
async function fetchAll() {
  try {
    await Promise.all([fetchEntries(), fetchWorkTypes(), fetchCustomersAndBookings(), fetchStats(), fetchTeamMembers()])
  } catch (err) {
    console.error('fetchAll error:', err)
  } finally {
    setLoading(false)
  }
}
```

### BUG-003: Child-records raderas utan business_id-filter i DELETE /api/projects
**Fil:** `app/api/projects/route.ts` rad 468–474
**Problem:** Alla child-deletes saknar `.eq('business_id', business.business_id)`. En autentiserad användare som känner till ett `project_id` tillhörande ett annat företag kan radera dess underdata (milestones, documents, materials, m.m.) utan att äga projektet.
**Symptom:** Kritisk data-förlust för drabbat företag. Syns inte i UI men data är borta.
**Rotorsak:** Säkerhetsfilter saknades vid implementationen.
**Fix:**
```typescript
// FEL:
await supabase.from('project_document').delete().eq('project_id', projectId)
await supabase.from('project_checklist').delete().eq('project_id', projectId)
// ... (alla child-deletes)

// RÄTT:
await supabase.from('project_document').delete()
  .eq('project_id', projectId).eq('business_id', business.business_id)
await supabase.from('project_checklist').delete()
  .eq('project_id', projectId).eq('business_id', business.business_id)
// ... (lägg till business_id på alla child-deletes)
```

### BUG-004 (ny): `.single()` utan null-check kraschar API på flera ställen
**Fil:** `app/api/quotes/route.ts` rad 32, 52, 149 · `app/api/projects/route.ts` rad 148, 200, 210, 229 · `app/dashboard/invoices/new/page.tsx` rad 94 · `app/dashboard/quotes/new/page.tsx` rad 365, 486
**Problem:** `.single()` kastar exception vid PGRST116 (ingen rad), men felhantering saknas eller är felaktig
**Symptom:** Server returnerar 500 istället för graceful error; UI-spinner fastnar
**Rotorsak:** Supabase `.single()` kastar vid tom resultatmängd utan guard
**Fix:**
```typescript
const { data: config, error } = await supabase
  .from('business_config')
  .select('default_payment_days')
  .eq('business_id', businessId)
  .single()

if (error?.code === 'PGRST116') {
  return NextResponse.json({ error: 'Config not found' }, { status: 404 })
}
if (error) {
  return NextResponse.json({ error: error.message }, { status: 500 })
}
```

**Fil:** `app/api/projects/route.ts` rad 469
**Problem:** `supabase.from('project_log').delete().eq('order_id', projectId)` — kolumnen heter troligen `project_id`, inte `order_id`. Projekt-loggar rensas aldrig vid projektradering.
**Symptom:** Orphaned projekt-loggar ligger kvar i databasen. Kan orsaka constraint-fel vid framtida migrationer.

**Fix:**
```typescript
// FEL:
await supabase.from('project_log').delete().eq('order_id', projectId)

// RÄTT:
await supabase.from('project_log').delete().eq('project_id', projectId)
```

### BUG-005: time_entry-kontroll saknar business_id i DELETE /api/projects
**Fil:** `app/api/projects/route.ts` rad 455–460
**Problem:** Kontrollen som förhindrar radering av projekt med tidrapporter filtrerar bara på `project_id`, inte `business_id`.
**Symptom:** Kan teoretiskt blockera borttagning av ett projekt om en annan aktör lyckas skicka in en time_entry med samma `project_id`.
**Rotorsak:** Saknat säkerhetsfilter.
**Fix:**
```typescript
// FEL:
const { count } = await supabase
  .from('time_entry')
  .select('*', { count: 'exact', head: true })
  .eq('project_id', projectId)

// RÄTT:
const { count } = await supabase
  .from('time_entry')
  .select('*', { count: 'exact', head: true })
  .eq('project_id', projectId)
  .eq('business_id', business.business_id)
```

### BUG-006: leads/route.ts GET-handler saknar try/catch
**Fil:** `app/api/leads/route.ts` rad 6–84
**Problem:** GET-funktionen har ingen try/catch. Om Supabase returnerar ett nätverksundantag (inte bara `error`-prop) kraschar Next.js runtime istället för att returnera 500.
**Symptom:** Okrypterat 500-svar utan JSON-body — kan krascha pipeline-sidan.
**Rotorsak:** Inkonsekvent felhantering jämfört med övriga routes.
**Fix:**
```typescript
// FEL:
export async function GET(request: NextRequest) {
  const business = await getAuthenticatedBusiness(request)
  // ... ingen try/catch

// RÄTT:
export async function GET(request: NextRequest) {
  try {
    const business = await getAuthenticatedBusiness(request)
    // ...
  } catch (error: any) {
    console.error('Get leads error:', error)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
}
```

### BUG-009 (ny): Quote-versioner utan `.order()` ger icke-deterministisk `.limit(1)`
**Fil:** `app/api/quotes/route.ts` rad 59–65
**Problem:** `.or(...)` utan `.order()` före `.limit(1)` — databas returnerar rader i odefinierad ordning
**Symptom:** Fel version returneras som "senaste" vid quote-versionshantering
**Rotorsak:** Explicit ORDER BY saknas
**Fix:**
```typescript
.or(`quote_id.eq.${parentQuoteId},parent_quote_id.eq.${parentQuoteId}`)
.order('version_number', { ascending: false })  // måste komma före limit
.limit(1)
```

### BUG-008: quote_items delete utan rollback-skydd i PUT /api/quotes
**Fil:** `app/api/quotes/route.ts` rad 566–588
**Problem:** Items raderas (`delete`) och sedan insertas nya. Om insert misslyckas (validation-fel, DB-constraint) är alla items borta utan möjlighet till återställning.
**Symptom:** Tom offert om backend-insert kraschar mitt i operationen.
**Rotorsak:** Supabase saknar inbyggda transaktioner via JS-klienten — delete+insert är inte atomär.
**Fix:** Flytta delete till efter lyckad insert, eller använd en Supabase RPC-funktion som kör hela operationen i en SQL-transaktion.

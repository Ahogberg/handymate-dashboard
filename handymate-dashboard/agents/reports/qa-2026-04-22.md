## Kritiska buggar

### BUG-001: DELETE projekt — fel kolumnnamn på project_log raderar aldrig loggar
**Fil:** `app/api/projects/route.ts` rad 469
**Prioritet:** Kritisk
**Problem:** `project_log`-tabellen rensas med `.eq('order_id', projectId)` men kolumnen heter `project_id` (konfirmerat via `sql/rot_rut_documents.sql` rad 80: `idx_project_log_project ON project_log(project_id)`). Supabase returnerar ingen felkod — delete kör mot 0 matchande rader och "lyckas". Projektloggar ackumuleras för evigt i databasen vid projektborttagning.
**Symptom:** Projektloggar raderas aldrig när ett projekt tas bort. Vid skalad användning orsakar detta databasuppsvällning. Om `order_id` råkar vara ett befintligt PK-värde någon annanstans i schemat kan fel rader raderas.
**Rotorsak:** Stavfel i kolumnnamn — `order_id` istället för `project_id`.
**Fix:**
```typescript
// FEL (rad 469):
await supabase.from('project_log').delete().eq('order_id', projectId)

// RÄTT:
await supabase.from('project_log').delete().eq('project_id', projectId)
```

### BUG-002: DELETE time_entry saknar business_id-filter — möjlig cross-tenant radering
**Fil:** `app/dashboard/time/page.tsx` rad 416
**Prioritet:** Kritisk
**Problem:** `handleDelete` skickar ett Supabase-klientanrop direkt utan `.eq('business_id', ...)` som defensivt lager. Anropet litar helt på Supabase RLS. Om RLS har ett hål (policy saknas, eller feature-flagga för bypass aktiveras) kan en autentiserad användare som känner till en annan `time_entry_id` radera den.
**Symptom:** Möjlig obehörig radering av annan businesses tidrapporter om RLS är felkonfigurerat eller inaktiverat.
**Rotorsak:** Klientoperationen saknar explicit business_id-filter som defensivt lager utöver RLS.
**Fix:**
```typescript
// FEL (rad 416):
const { error } = await supabase.from('time_entry').delete().eq('time_entry_id', id)

// RÄTT:
const { error } = await supabase
  .from('time_entry')
  .delete()
  .eq('time_entry_id', id)
  .eq('business_id', business.business_id)
```

### BUG-003: GET /api/leads saknar try/catch — uncaught exceptions crashar Next.js
**Fil:** `app/api/leads/route.ts` rad 6–85
**Prioritet:** Kritisk
**Problem:** GET-handlern har inget `try/catch`. `Promise.all([query, statsRes])` kan kasta om Supabase är nere eller om DB-schemat avviker från vad koden förväntar sig. Next.js returnerar då en generisk 500 utan `{ error: ... }` i body. Pipeline-sidan (`/dashboard/pipeline`) som anropar `/api/leads` (indirekt) hanterar inte detta och visar tomt kanban-bräde utan felmeddelande.
**Symptom:** Pipeline laddas inte; användaren ser tomt bräde utan förklaring.
**Rotorsak:** Inkonsekvent felhantering — PATCH och POST har korrekt try/catch men GET saknar wrapper.
**Fix:**
```typescript
// FEL: GET-funktionen saknar try/catch (rad 6–85)
export async function GET(request: NextRequest) {
  const business = await getAuthenticatedBusiness(request)
  if (!business) { ... }
  // direkt kod utan felhantering

// RÄTT:
export async function GET(request: NextRequest) {
  try {
    const business = await getAuthenticatedBusiness(request)
    if (!business) {
      return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    }
    // ... resten av koden
    return NextResponse.json({ leads: ..., stats: ... })
  } catch (error: any) {
    console.error('Get leads error:', error)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
}
```

### BUG-004: Pipeline skickar business_id som öppen URL-param — säkerhetsrisk
**Fil:** `app/dashboard/pipeline/page.tsx` rad 481, 496, 507
**Prioritet:** Kritisk
**Problem:** `business_id` skickas som URL-query-parameter (`/api/pipeline?business_id=...`, `/api/pipeline/activity?business_id=...`, `/api/pipeline/deals/${dealId}/activity?business_id=...`). Om API-routerna litar på URL-parametern utan att verifiera mot den autentiserade sessionen kan vilken inloggad användare byta ut sin `business_id` mot ett annat företags ID och läsa deras pipeline-data.
**Symptom:** Potentiell informationsläcka — autentiserad användare kan läsa annan businesses pipeline om API-routen inte kör `getAuthenticatedBusiness()` och istället litar på query-param.
**Rotorsak:** Business-kontexten skickas som öppen URL-parameter istället för att API:et hämtar den via auth-cookie.
**Fix:**
```typescript
// FEL (rad 481):
const res = await fetch(`/api/pipeline?business_id=${business.business_id}`)

// RÄTT — ta bort business_id från URL, låt API hämta via getAuthenticatedBusiness:
const res = await fetch('/api/pipeline')
// API-routen måste använda: const business = await getAuthenticatedBusiness(request)
```

### BUG-005: "Från offert"-knapp navigerar bort och förlorar fakturadata
**Fil:** `app/dashboard/invoices/new/page.tsx` rad 403
**Prioritet:** Viktig
**Problem:** Knappen "Från offert" anropar `router.push('/dashboard/quotes')` vilket navigerar bort från fakturaformuläret. All inmatad data (kund, rader, datum) förloras. Intentionen är sannolikt att öppna en modal eller navigera tillbaka med `?fromQuote=...` som query-param.
**Symptom:** Användare som fyllt i fakturarader klickar "Från offert", förlorar all data och hamnar på offertlistan utan möjlighet att komma tillbaka till fakturaformuläret med bevarad data.
**Rotorsak:** Implementationen är ofärdig — borde vara en modal eller länk med `returnUrl`.
**Fix:** Implementera en offert-väljarmodal, eller navigera med return-URL:
```typescript
// FEL (rad 403):
onClick={() => router.push('/dashboard/quotes')}

// RÄTT — exempelfix med query-param:
onClick={() => router.push(`/dashboard/quotes?selectForInvoice=true&returnUrl=${encodeURIComponent('/dashboard/invoices/new')}`)}
// Alternativt: öppna modal med offertlista direkt i sidan
```

### BUG-006: customer_activity .single() utan guard — swallowear databasfel tyst
**Fil:** `app/dashboard/pipeline/page.tsx` rad 827
**Prioritet:** Viktig
**Problem:** `fetchCustomerEnrichment` kör `.single()` mot `customer_activity` utan att kontrollera att exakt en rad finns. Supabase returnerar error `PGRST116` (0 rows) när inga rader hittas. `catch`-blocket sätter `lastContact(null)` och döljer felet — men detta döljer också verkliga databasfel (t.ex. om tabellen `customer_activity` inte existerar eller har RLS-problem).
**Symptom:** "Senaste kontakt" visas aldrig i deal-panelen ens om data finns — faktiska fel syns inte.
**Rotorsak:** `.single()` kastas mot en fråga som kan returnera 0 rader; bör använda `.maybeSingle()`.
**Fix:**
```typescript
// FEL (rad 827):
.limit(1)
.single()

// RÄTT:
.limit(1)
.maybeSingle()
```

### BUG-007: Canvas auto-save-timer fångar stale saveCanvas-closure
**Fil:** `components/project/ProjectCanvas.tsx` rad 110–127
**Prioritet:** Viktig
**Problem:** Auto-save-timern (setInterval 30s) skapas i canvas-init-effekten med `[fabricLoaded]` som dependency. `saveCanvas` är definierad med `useCallback([resolvedType, resolvedId])` men fångas som closure när timern skapas. Om `resolvedId` ändras (SPA-navigering utan full unmount) sparas canvasen mot fel `entityId` tills nästa `fabricLoaded`-cykel — potentiellt 30 sekunder med fel entity-sparning.
**Symptom:** Vid SPA-navigering mellan projekt kan canvas-data sparas till fel projektets entity_id i upp till 30 sekunder.
**Rotorsak:** Auto-save-timer fångar `saveCanvas` via closure vid skapandetillfället; ändring av `resolvedId` uppdaterar inte timern.
**Fix:**
```typescript
// Använd ref för att alltid ha senaste saveCanvas:
const saveCanvasRef = useRef(saveCanvas)
useEffect(() => { saveCanvasRef.current = saveCanvas }, [saveCanvas])

// I timer-init-effekten (rad 110):
autoSaveTimerRef.current = setInterval(() => {
  saveCanvasRef.current()  // alltid senaste version
}, 30000)
```

### BUG-008: fetchFormSubmissions / fetchFormTemplates anropas aldrig korrekt vid tab-byte

**Prioritet:** Viktig
**Problem:** `fetchFormSubmissions` och `fetchFormTemplates` definieras som vanliga `async function` (inte `useCallback`) och anropas i useEffect vid `activeTab === 'checklists'`. De ingår inte i useEffect dependency-array och är potentiellt stale closures — de fångar `projectId` vid definitionstillfället men om komponenten re-renderas av annan anledning innan tab-bytet kan gamla `projectId` användas.

const fetchFormSubmissions = useCallback(async () => {

}, [])

### BUG-009: handleDeleteReport på vehicle reports swallowear fel — lokal state uppdateras utan bekräftelse
**Fil:** `app/dashboard/vehicles/page.tsx` rad 186–198
**Prioritet:** Viktig
**Problem:** `handleDeleteReport` anropar `/api/vehicle-reports` med DELETE men kontrollerar inte `res.ok`. Oavsett API-svar körs `setReports(prev => prev.filter(...))` och raden försvinner från UI. Om backend returnerar 4xx/5xx visas inga felmeddelanden och raden dyker upp igen vid nästa reload.
**Symptom:** Användaren klickar "Ta bort", raden försvinner i UI, men efter omladdning är den tillbaka — inga felmeddelanden visas.
**Rotorsak:** Saknad `res.ok`-kontroll; optimistisk state-uppdatering utan felhantering.
**Fix:**
```typescript
// FEL (rad 186-198):
const handleDeleteReport = async (reportId: string) => {
  if (!confirm('Ta bort denna körrapport?')) return
  try {
    await fetch('/api/vehicle-reports', {
      method: 'DELETE',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ id: reportId }),
    })
    setReports(prev => prev.filter(r => r.id !== reportId))
  } catch (e) {
    console.error('Delete report error:', e)
  }
}

// RÄTT:
const handleDeleteReport = async (reportId: string) => {
  if (!confirm('Ta bort denna körrapport?')) return
  try {
    const res = await fetch('/api/vehicle-reports', {
      method: 'DELETE',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ id: reportId }),
    })
    if (!res.ok) throw new Error('Radering misslyckades')
    setReports(prev => prev.filter(r => r.id !== reportId))
  } catch (e) {
    console.error('Delete report error:', e)
    alert('Kunde inte ta bort körrapport')
  }
}
```

### BUG-010: TeamPageContent — dynamisk import saknar ssr: false
**Fil:** `app/dashboard/settings/page.tsx` rad 46–52
**Prioritet:** Mindre
**Problem:** `TeamPageContent` importeras dynamiskt utan `ssr: false`. Team-sidan är en `'use client'`-komponent som använder `useEffect` och browser-API — om Next.js försöker server-rendera den kan SSR-fel eller hydreringsproblem uppstå.
**Symptom:** Potentiell hydreringsfelmatchning eller "window is not defined"-fel vid SSR om team-sidan använder window-API.
**Fix:**
```typescript
// FEL:
const TeamPageContent = dynamic(() => import('@/app/dashboard/team/page'), {
  loading: () => (...)
})

// RÄTT:
const TeamPageContent = dynamic(() => import('@/app/dashboard/team/page'), {
  loading: () => (...),
  ssr: false,
})
```

### BUG-011: Rate limit store är in-memory per process — resettas vid varje Vercel cold start
**Fil:** `lib/auth.ts` rad 146–156
**Prioritet:** Mindre
**Problem:** `rateLimitStore` är en `Map` i modulens globala scope. Vercel kör varje request i potentiellt skilda serverless-instanser med cold starts. Rate-limit-räknaren nollställs vid varje ny instans — rate limiting skyddar inte mot distribuerade angrepp. Kommentaren på rad 145 nämner detta ("I produktion bör detta vara Redis") men inga åtgärder har vidtagits.
**Symptom:** Angripare kan skicka fler SMS/AI-anrop än tillåtet genom att sprida requests över tid eller tvinga cold starts.
**Fix:** Implementera Redis-baserad rate limiting (t.ex. Upstash KV på Vercel) eller dokumentera tydligt att rate limiting enbart är ett mjukt skydd.

### BUG-012: Extensiv användning av `any` utan typguards i projektsidans dataflöden
**Fil:** `app/dashboard/projects/[id]/page.tsx` rad 499–519 (state-deklarationer)
**Prioritet:** Mindre
**Problem:** `documents`, `logs`, `checklists`, `checklistTemplates`, `workOrders`, `formSubmissions`, `formTemplates`, `aiLogs` är alla deklarerade som `any[]`. TypeScript-skyddet är borta för dessa typer — null-fel och fältmisstagningar fångas inte vid kompilering.
**Symptom:** Runtime-krasch om API:et returnerar oväntat format (t.ex. `checklists` är `null` istället för `[]`; `.map()` på `null` kraschar utan TypeScript-varning).
**Fix:** Definiera interfaces för dessa entiteter eller använd `unknown[]` + typeguards. Minst: sätt `useState<SomeType[]>([])` med explicit interface.

### BUG-013: Canvas-sparning swallowear API-fel tyst utan återkoppling

// Nuvarande (rad 185):
} catch { /* ignore */ }

} catch {

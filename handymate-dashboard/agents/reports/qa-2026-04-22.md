## Kritiska buggar

### BUG-001: DELETE projekt — felaktig kolumnreferens i project_log-query
**Fil:** `app/api/projects/route.ts` rad 469
**Problem:** Vid radering av projekt körs `.eq('order_id', projectId)` mot tabellen `project_log`, men kolumnen heter sannolikt `project_id`. `order_id` är inte en dokumenterad kolumn i `project_log`.
**Symptom:** Projektradering misslyckas INTE synligt (Supabase ignorerar delete mot felaktig kolumn utan FK-constraint), men `project_log`-rader lämnas kvar i databasen. Om `project_log` har en FK på `project_id` kraschar hela DELETE med 500.
**Rotorsak:** Copy-paste-fel — troligen kopierat från ett `order`-relaterat flöde och kolumnnamnet ändrades aldrig.
**Fix:**
```typescript
// FEL (rad 469):
await supabase.from('project_log').delete().eq('order_id', projectId)

// RÄTT:
await supabase.from('project_log').delete().eq('project_id', projectId)
```

### BUG-002: DELETE projekt — saknar business_id-filter på child-tabeller (säkerhetshål)
**Fil:** `app/api/projects/route.ts` rad 468–474
**Problem:** Alla child-tabell-raderingar saknar `.eq('business_id', business.business_id)`. `project_id` kontrolleras mot `project`-tabellen med `business_id` i sista steget, men en angripare kan skicka ett `projectId` som tillhör ett annat företag för att radera dess data.
**Symptom:** En inloggad användare från företag A kan radera `project_document`, `project_checklist`, `project_assignment` etc. från företag B om de känner till ett `project_id`.
**Rotorsak:** Bristande auktorisering — business_id-filtret läggs bara på huvud-radering, inte på child-raderingar.
**Fix:**
```typescript
// FEL:
await supabase.from('project_document').delete().eq('project_id', projectId)
// ... (inga business_id-filter)

// RÄTT — verifiera ägarskap INNAN child-raderingar:
const { data: projectCheck } = await supabase
  .from('project')
  .select('project_id')
  .eq('project_id', projectId)
  .eq('business_id', business.business_id)
  .single()

if (!projectCheck) {
  return NextResponse.json({ error: 'Not found' }, { status: 404 })
}
// Nu är det säkert att radera children
await supabase.from('project_document').delete().eq('project_id', projectId)
// ...
```

### BUG-003: GET /api/leads — saknar try/catch, okontrollerade fel ger 500 utan loggning
**Fil:** `app/api/leads/route.ts` rad 6–85
**Problem:** GET-handleren har ingen try/catch. Om ett Supabase-anrop kastar (nätverksfel, timeout, etc.) bubblar felet okontrollerat och Next.js returnerar en ogenomskådlig 500.
**Symptom:** Användaren ser ett tomt pipeline-gränssnitt. Inga loggar i Vercel. Alla andra API-routes (customers, projects, quotes) har try/catch men leads GET saknar det.
**Rotorsak:** Konsekvensbrist i felhantering.
**Fix:**
```typescript
// FEL — ingen try/catch:
export async function GET(request: NextRequest) {
  const business = await getAuthenticatedBusiness(request)
  // ... direkt Supabase-anrop utan felhantering

// RÄTT:
export async function GET(request: NextRequest) {
  try {
    const business = await getAuthenticatedBusiness(request)
    if (!business) {
      return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    }
    // ... övrig logik
    return NextResponse.json({ leads: ..., stats: ... })
  } catch (error: any) {
    console.error('Get leads error:', error)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
}
```

### BUG-004: PUT /api/projects — .or()-query med potentiella null-värden kraschar pipeline-flytt
**Fil:** `app/api/projects/route.ts` rad 413
**Problem:** `.or(`quote_id.eq.${project.quote_id},lead_id.eq.${project.lead_id}`)` exekveras utan kontroll på om `quote_id` eller `lead_id` är null. PostgREST tolkar `quote_id.eq.null` som en strängjämförelse, inte NULL-check, vilket ger felaktiga resultat.
**Symptom:** Vid projektavslut utan kopplad offert eller lead hittas fel deal eller kastas ett exception som suppressas men skräpar i loggarna.
**Rotorsak:** Saknad null-guard för optional foreign keys.
**Fix:**
```typescript
// FEL (rad 409–415):
const { data: linkedDeal } = await supabase
  .from('deal')
  .select('id')
  .eq('business_id', business.business_id)
  .or(`quote_id.eq.${project.quote_id},lead_id.eq.${project.lead_id}`)
  .maybeSingle()

// RÄTT:
const orParts: string[] = []
if (project.quote_id) orParts.push(`quote_id.eq.${project.quote_id}`)
if (project.lead_id) orParts.push(`lead_id.eq.${project.lead_id}`)

let linkedDeal = null
if (orParts.length > 0) {
  const { data } = await supabase
    .from('deal')
    .select('id')
    .eq('business_id', business.business_id)
    .or(orParts.join(','))
    .maybeSingle()
  linkedDeal = data
}
```

### BUG-005: vehicles/page.tsx — useEffect på veckonavigering saknar business_id-guard
**Fil:** `app/dashboard/vehicles/page.tsx` rad 156–158
**Problem:** Det andra `useEffect`-blocket triggas på `weekDate`/`selectedVehicle`-ändringar och kallar `fetchReports()` utan att kolla `business.business_id`. `fetchReports` använder `useCallback` som fångar `week.start`/`week.end` men inte `business` — om business-contexten inte laddats klart kan API-anropet göras för tidigt.
**Symptom:** På mobilenheter med långsam uppstart kan rapportlistan se tom ut initialt och aldrig reladda.
**Rotorsak:** Dependency array inkluderar inte `business?.business_id`.
**Fix:**
```typescript
// FEL:
useEffect(() => {
  if (!loading) fetchReports()
}, [weekDate, selectedVehicle])

// RÄTT:
useEffect(() => {
  if (!loading && business?.business_id) fetchReports()
}, [weekDate, selectedVehicle, business?.business_id])
```

### BUG-007: ProjectCanvas — canvas history-refs rensas inte vid component cleanup
**Fil:** `components/project/ProjectCanvas.tsx` rad 122–127
**Problem:** `useEffect` cleanup-funktionen kör `canvas.dispose()` och rensar `autoSaveTimerRef`, men rengör inte `historyRef.current` och `historyIndexRef.current`. Vid remount (t.ex. tab-switch i projektdetaljer) ackumuleras gamla history-states.
**Symptom:** Undo/Redo beter sig oförutsägbart efter att canvas monterats om — kan hoppa till states från ett tidigare session.
**Fix:**
```typescript
// FEL — cleanup saknar historyreset:
return () => {
  if (autoSaveTimerRef.current) clearInterval(autoSaveTimerRef.current)
  window.removeEventListener('resize', handleResize)
  canvas.dispose()
}

// RÄTT:
return () => {
  if (autoSaveTimerRef.current) clearInterval(autoSaveTimerRef.current)
  window.removeEventListener('resize', handleResize)
  canvas.dispose()
  historyRef.current = []
  historyIndexRef.current = -1
}
```

### BUG-008: api/leads/route.ts PATCH — aktivitetsloggning är blocking och saknar felhantering
**Fil:** `app/api/leads/route.ts` rad 128–138
**Problem:** `await supabase.from('lead_activities').insert(...)` körs blocking utan try/catch. Om denna insert misslyckas (tabellen saknas, constraint-fel) kastas ett exception som propagerar och ger 500 — trots att huvud-uppdateringen av lead lyckades.
**Symptom:** Statusändring på lead misslyckas med 500 om `lead_activities`-tabellen inte finns i produktion.
**Fix:**
```typescript
// FEL — blocking utan felhantering:
if (updates.status) {
  const actId = 'la_' + ...
  await supabase.from('lead_activities').insert({...})
}
return NextResponse.json({ lead: data })

// RÄTT — non-blocking:
if (updates.status) {
  try {
    const actId = 'la_' + ...
    await supabase.from('lead_activities').insert({...})
  } catch { /* non-blocking logging */ }
}
return NextResponse.json({ lead: data })
```

### BUG-009: api/projects/route.ts PUT — existingProject hämtas utan business_id-filter
**Fil:** `app/api/projects/route.ts` rad 293–296
**Problem:** Vid 4-eyes-check saknar `existingProject`-hämtningen `.eq('business_id', business.business_id)`. En angripare kan sätta `project_id` till ett annat företags projekt och läsa dess `budget_amount`.
**Symptom:** Informationsläckage — `budget_amount` från ett annat företags projekt exponeras och kan trigga pending_approvals-insert i det egna företaget med felaktig data.
**Fix:**
```typescript
// FEL (rad 292–296):
const { data: existingProject } = await supabase
  .from('project')
  .select('budget_amount')
  .eq('project_id', project_id)
  .single()

// RÄTT:
const { data: existingProject } = await supabase
  .from('project')
  .select('budget_amount')
  .eq('project_id', project_id)
  .eq('business_id', business.business_id)
  .single()
```

### BUG-011: lib/auth.ts — in-memory rate limit fungerar inte i Vercel serverless
**Fil:** `lib/auth.ts` rad 146–156
**Problem:** `rateLimitStore` är en `Map` i modulens global scope. I Vercel serverless-miljö körs requests i separata Lambda-instanser — rate limit-state delas aldrig mellan dem.
**Symptom:** En klient kan göra obegränsat antal SMS/AI-anrop per minut i produktion; rate limiting är bara effektivt under lokal utveckling.
**Rotorsak:** Arkitekturellt — in-memory state är per-instans i serverless.
**Rekommendation:** Ersätt med Upstash Redis eller Supabase-baserad rate limiting för att fungera i produktion.

### BUG-012: ProjectCanvas — canvas-konstruktorn saknar try/catch, fabricError visas aldrig
**Fil:** `components/project/ProjectCanvas.tsx` rad 76–83
**Problem:** `new fabric.Canvas(...)` anropas utan try/catch. Om konstruktorn kastar (t.ex. `canvasRef.current` är null pga race condition) propagerar felet som ett React-render-fel istället för att visa det vänliga `fabricError`-meddelandet.
**Symptom:** White screen i stället för "Skissblocket kunde inte laddas".
**Fix:**
```typescript
// FEL — ingen felhantering runt canvas-konstruktorn:
const canvas = new fabric.Canvas(canvasRef.current, { ... })

// RÄTT:
let canvas: any
try {
  canvas = new fabric.Canvas(canvasRef.current, { ... })
} catch (err) {
  console.error('[ProjectCanvas] Canvas init failed:', err)
  setFabricError(true)
  setLoading(false)
  return
}
```

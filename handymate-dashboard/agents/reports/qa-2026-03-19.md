### BUG-001: DELETE projekt — fel kolumnnamn på project_log suddar data
**Fil:** `app/api/projects/route.ts` rad 346
**Problem:** `project_log`-tabellen rensas med `.eq('order_id', projectId)` men kolumnen heter med stor sannolikhet `project_id`, inte `order_id`. Felet är stumt — Supabase returnerar ingen rad om kolumnen inte matchar, men delete-operationen lyckas (påverkar 0 rader). Vid nästa delete av ett annat projekt med ett verkligt `order_id`-fält kan fel rader raderas.
**Symptom:** Projektloggar rensas inte vid projektborttagning (data läcker kvar i DB). Om `order_id` råkar matcha ett annat projekts pk-värde raderas fel logg.
**Rotorsak:** Fel kolumnnamn har skrivits i delete-kedjan.
**Fix:**
```typescript
// FEL (rad 346):
await supabase.from('project_log').delete().eq('order_id', projectId)

// RÄTT:
await supabase.from('project_log').delete().eq('project_id', projectId)
```

### BUG-002: DELETE time_entry saknar business_id-filter — valfri rad kan tas bort
**Fil:** `app/dashboard/time/page.tsx` rad 416
**Problem:** `handleDelete` skickar direkt ett Supabase-anrop från klienten utan `.eq('business_id', ...)` som guard. Eftersom detta är ett klientanrop (inte via API-route med `getAuthenticatedBusiness`) räcker det att en angripare tar en `time_entry_id` från ett annat företag och skickar anropet — RLS i Supabase kan stoppa det, men om RLS av någon anledning är felkonfigurerat raderas raden.
**Symptom:** Möjlig obehörig radering av annan business tidrapporter om Supabase RLS inte täcker detta scenario.
**Rotorsak:** Klientoperationen saknar explicit `business_id`-filter som defensivt lager.
**Fix:**
```typescript
// FEL (rad 416):
const { error } = await supabase.from('time_entry').delete().eq('time_entry_id', id)

// RÄTT:
const { error } = await supabase
  .from('time_entry')
  .delete()
  .eq('time_entry_id', id)
  .eq('business_id', business.business_id)
```

### BUG-003: GET /api/leads saknar try/catch — uncaught exceptions ger 500 utan body
**Fil:** `app/api/leads/route.ts` rad 6–84
**Problem:** `GET`-handlern har inget `try/catch`. Om `supabase`-anropet eller `Promise.all` kastar ett undantag kraschar Next.js och returnerar en generisk 500 utan `{ error: ... }` i body. Pipeline-sidan (`/dashboard/pipeline`) hanterar inte detta och visar en tom pipeline utan felmeddelande.
**Symptom:** Pipeline laddas inte, användaren ser tomt kanban-bräde utan förklaring.
**Rotorsak:** Inkonsekvent felhantering — PATCH och POST har korrekt struktur men GET saknar wrapper.
**Fix:**
```typescript
// FEL: GET-funktionen har ingen try/catch
export async function GET(request: NextRequest) {
  const business = await getAuthenticatedBusiness(request)
  // ... direkt kod utan try/catch

// RÄTT:
export async function GET(request: NextRequest) {
  try {
    const business = await getAuthenticatedBusiness(request)
    if (!business) {
      return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    }
    // ... resten av koden
  } catch (error: any) {
    console.error('Get leads error:', error)
    return NextResponse.json({ error: error.message }, { status: 500 })
  }
}
```

### BUG-004: Pipeline skickar business_id som query-param i URL — exponerar intern ID
**Fil:** `app/dashboard/pipeline/page.tsx` rad 463, 478, 488, 768
**Problem:** `business_id` skickas som URL-query-parameter (`/api/pipeline?business_id=...`). API:et bör hämta business_id via `getAuthenticatedBusiness()` från cookien — inte från URL-parametern. Om API-routen litar på URL-parametern kan vilken autentiserad användare byta ut sin `business_id` mot ett annat företags ID och läsa deras pipeline.
**Symptom:** Möjlig informationsläcka om `/api/pipeline` inte validerar att query-params `business_id` matchar den autentiserade sessionen.
**Rotorsak:** Business-kontexten på klientsidan används korrekt, men att skicka den som öppen URL-param är fel mönster — API:et ska alltid verifiera via auth-lagret.
**Fix:**
```typescript
// FEL (rad 463):
const res = await fetch(`/api/pipeline?business_id=${business.business_id}`)

// RÄTT — ta bort business_id från URL, låt API:et hämta det via getAuthenticatedBusiness:
const res = await fetch('/api/pipeline')
// API-routen bör använda: const business = await getAuthenticatedBusiness(request)
```

### BUG-006: CustomerActivity .single() utan guard — kraschar om ingen rad finns (pipeline)
**Fil:** `app/dashboard/pipeline/page.tsx` rad 803–810
**Problem:** `fetchCustomerEnrichment` kör `.single()` mot `customer_activity` utan att hantera fallet att inga rader finns. Supabase returnerar error `PGRST116` (0 rows) som fångas av `catch`-blocket — men `catch` sätter `lastContact(null)` och swallowear felet. Det fungerar men är ett mönster som döljer faktiska databasfel bakom tysta missar.
**Symptom:** Senaste kontakt visas aldrig i deal-panelen ens om data finns — om tabellen `customer_activity` inte existerar eller har ett annat schema kraschar det tyst.
**Rotorsak:** `.single()` kastas felaktigt mot en fråga som kan returnera 0 rader; bör använda `.maybeSingle()`.
**Fix:**
```typescript
// FEL (rad 803):
const { data: activity } = await supabase
  .from('customer_activity')
  .select('activity_type, created_at')
  .eq('customer_id', customerId)
  .order('created_at', { ascending: false })
  .limit(1)
  .single()

// RÄTT:
const { data: activity } = await supabase
  .from('customer_activity')
  .select('activity_type, created_at')
  .eq('customer_id', customerId)
  .order('created_at', { ascending: false })
  .limit(1)
  .maybeSingle()
```

### BUG-007: Canvas auto-save-timer startar vid varje tool/color-ändring p.g.a. saknad dependency
**Fil:** `components/project/ProjectCanvas.tsx` rad 110–127
**Problem:** `useEffect` som initierar canvas har `eslint-disable-line react-hooks/exhaustive-deps` på kommentar och beror på `[fabricLoaded]`. Auto-save-timern (setInterval 30s) skapas korrekt och rensas i cleanup. Men `saveCanvas`-funktionen definieras med `useCallback([resolvedType, resolvedId])` och används inne i timern via closures. Om `resolvedId` ändras (t.ex. navigering utan full unmount) sparas canvasen mot fel entitet tills nästa `fabricLoaded`-cykel.
**Symptom:** Vid SPA-navigering mellan projekt kan canvas-data sparas till fel `entityId` under de 30 sekunder som löper kvar i timern.
**Rotorsak:** Auto-save-timern skapas i canvas-init-effekten men fångar `saveCanvas`-closure vid det tillfälle den skapades. Ändring av `resolvedId` uppdaterar inte timern.
**Fix:**
```typescript
// Använd en ref för saveCanvas för att alltid ha den senaste versionen:
const saveCanvasRef = useRef(saveCanvas)
useEffect(() => { saveCanvasRef.current = saveCanvas }, [saveCanvas])

// I timer-init-effekten:
autoSaveTimerRef.current = setInterval(() => {
  saveCanvasRef.current()
}, 30000)
```

### BUG-008: `fetchFormSubmissions` och `fetchFormTemplates` anropas aldrig vid tab-byte
**Fil:** `app/dashboard/projects/[id]/page.tsx` rad 616–645
**Problem:** I `useEffect` på `activeTab` anropas `fetchChecklists()`, `fetchChecklistTemplates()`, `fetchFormSubmissions()` och `fetchFormTemplates()` vid `activeTab === 'checklists'`. Men `fetchFormSubmissions` och `fetchFormTemplates` är definierade längre ner (rad 770+) och är deklarerade som vanliga `async function` (inte `useCallback`). De saknas i `useEffect`s dependency-array men detta är inte problemet — problemet är att de inte är deklarerade med `useCallback` och kan vara stale closures. Deras deklarationer nedan i filen körs, men de kan referera till stale `project` eller `supabase`-state.
**Symptom:** Formulärmallar och inskickade formulär kan visa gammal data eller laddas inte alls efter projektet uppdaterats.
**Rotorsak:** `fetchFormSubmissions`/`fetchFormTemplates` definieras inte med `useCallback` och ingår inte i `useEffect`s dependency-array.
**Fix:**
```typescript
// Konvertera till useCallback eller flytta definitionen ovanför useEffect:
const fetchFormSubmissions = useCallback(async () => {
  // befintlig kod
}, [projectId])

const fetchFormTemplates = useCallback(async () => {
  // befintlig kod
}, [])

// Lägg till i useEffect dependency-array:
}, [activeTab, fetchProjectTeam, fetchFormSubmissions, fetchFormTemplates])
```

### BUG-009: `handleDelete` på vehicle reports swallowear fel — ingen feedback vid misslyckad radering
**Fil:** `app/dashboard/vehicles/page.tsx` rad 186–198
**Problem:** `handleDeleteReport` anropar `/api/vehicle-reports` med DELETE men kontrollerar inte `res.ok`. Om API:et returnerar 4xx/5xx tas raden ändå bort från lokal state (`setReports(prev => prev.filter(...))`), men backend-datan är orörd. Vid nästa reload dyker raden upp igen utan att användaren förstår varför.
**Symptom:** Användaren klickar "Ta bort", raden försvinner i UI, men efter omladdning är den tillbaka — inga felmeddelanden visas.
**Rotorsak:** Saknad `res.ok`-kontroll och lokal state uppdateras optimistiskt utan bekräftelse.
**Fix:**
```typescript
// FEL (rad 186-198):
const handleDeleteReport = async (reportId: string) => {
  if (!confirm('Ta bort denna körrapport?')) return
  try {
    await fetch('/api/vehicle-reports', {
      method: 'DELETE',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ id: reportId }),
    })
    setReports(prev => prev.filter(r => r.id !== reportId))
  } catch (e) {
    console.error('Delete report error:', e)
  }
}

// RÄTT:
const handleDeleteReport = async (reportId: string) => {
  if (!confirm('Ta bort denna körrapport?')) return
  try {
    const res = await fetch('/api/vehicle-reports', {
      method: 'DELETE',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ id: reportId }),
    })
    if (!res.ok) throw new Error('Radering misslyckades')
    setReports(prev => prev.filter(r => r.id !== reportId))
  } catch (e) {
    console.error('Delete report error:', e)
    alert('Kunde inte ta bort körrapport')
  }
}
```

### BUG-013: Rate limit store är in-memory per process — nollställs vid varje Vercel-deployment
**Fil:** `lib/auth.ts` rad 146–156
**Problem:** `rateLimitStore` är en `Map` i modulens globala scope. På Vercel körs varje request i potentiellt olika serverless-instanser (cold starts), vilket innebär att rate-limit-räknaren nollställs vid varje ny instance. Rate limiting skyddar alltså inte mot distribuerade angrepp.
**Symptom:** Angripare kan skicka fler SMS/AI-anrop än tillåtet genom att sprida requests över tid eller tvinga cold starts.
**Rotorsak:** In-memory store fungerar inte i stateless serverless-miljö. Kommentaren på rad 145 nämner detta ("I produktion bör detta vara Redis") men ingen åtgärd har vidtagits.
**Fix:** Implementera Redis-baserad rate limiting (t.ex. via Upstash) eller Vercel KV. Detta är ett arkitekturellt problem — kortsiktig fix är att dokumentera begränsningen tydligare.
